## 1. Executive Summary

mcp-proxy is a command-line tool written in Go that acts as an authentication proxy for MCP (Model Context Protocol) servers. It enables developers to use MCP servers that require OAuth2.1 authentication by handling the authentication flow transparently. The tool exposes a local stdio interface while connecting to remote MCP servers via streamable HTTP, automatically managing OAuth2.1 tokens including refresh and re-authentication when needed.

**Target Audience:** Developers who need to interact with OAuth2.1-protected MCP servers through stdio-based clients (e.g., Claude Desktop, Claude Code).

**Key Value Proposition:** Eliminates the need for developers to manually handle OAuth2.1 authentication flows when using MCP servers, providing seamless token management with automatic refresh and secure local caching.

## 3. Scope

### 3.1 In Scope
- CLI tool with stdio interface for local MCP clients
- OAuth2.1 authentication flow with PKCE support
- Automatic token refresh using refresh_token
- Token caching in local filesystem (~/.cache/mcp-proxy/)
- Connection to remote MCP servers via streamable HTTP
- Automatic retry after authentication
- Support for environment variable-based credential configuration
- HTTPS enforcement for MCP server URLs
- OpenTelemetry tracing with JSONL output
- Binary distribution via GitHub Releases (Linux, macOS, Windows)

### 3.2 Out of Scope (Non-Goals)
- Support for authentication methods other than OAuth2.1 (API keys, basic auth, etc.)
- Support for MCP server transports other than streamable HTTP (SSE, stdio)
- GUI or web interface
- Multi-user token management
- Token encryption at rest (file system permissions are sufficient)
- Supporting multiple MCP servers simultaneously in a single process
- Homebrew tap or go install distribution (deferred to future versions)
- Token revocation or logout functionality

## 4. User Personas & Actors

**Primary Actor: Developer**
- Uses MCP servers that require OAuth2.1 authentication
- Works with stdio-based MCP clients (Claude Desktop, Claude Code, custom tools)
- Needs seamless authentication without manual token management
- Values simplicity and reliability
- Comfortable with command-line tools
- May work with multiple MCP servers across different projects

## 5. Usage Scenarios

### SC-001: First-time use (no cached token)
**Actor:** Developer

**Preconditions:**
- No cached token file exists for the target MCP server URL
- Developer has valid client_id and client_secret (via CLI flags or environment variables)
- MCP server supports OAuth2.1 with .well-known discovery

**Flow:**
1. Developer runs: `mcp-proxy -u https://mcp.example.com`
2. mcp-proxy validates client_id and client_secret are not empty
3. mcp-proxy exposes stdio interface locally for MCP client to connect
4. mcp-proxy connects to remote MCP server at https://mcp.example.com via streamable HTTP
5. MCP server responds with authentication required (via .well-known mechanism)
6. mcp-proxy discovers OAuth2.1 endpoints via .well-known/oauth-authorization-server
7. mcp-proxy starts temporary HTTP server on localhost:3000 for OAuth callback
8. mcp-proxy generates PKCE code_verifier and code_challenge
9. mcp-proxy opens system browser to OAuth2.1 authorization URL with callback http://localhost:3000/oauth2callback
10. User completes OAuth flow in browser (login, consent)
11. Browser redirects to localhost:3000/oauth2callback with authorization code
12. mcp-proxy exchanges authorization code for access_token and refresh_token using PKCE code_verifier
13. mcp-proxy saves tokens to ~/.cache/mcp-proxy/{url_base64}.json with expiration time and file permissions 0600
14. mcp-proxy stops temporary HTTP server
15. mcp-proxy retries the original MCP server request with access_token via streamable HTTP
16. MCP server responds successfully
17. mcp-proxy forwards response to local client via stdio

**Postconditions:**
- Token file exists at ~/.cache/mcp-proxy/{url_base64}.json with permissions 0600
- access_token, refresh_token, and expiration_time are stored
- MCP client receives successful response
- Temporary HTTP server is stopped

**Exceptions:**
- **EXC-001a**: client_id is empty → mcp-proxy exits with error: "Error: client_id is required. Set via --client-id flag or GOOGLE_CLIENT_ID environment variable."
- **EXC-001b**: client_secret is empty → mcp-proxy exits with error: "Error: client_secret is required. Set via --client-secret flag or GOOGLE_CLIENT_SECRET environment variable."
- **EXC-001c**: MCP server URL is not HTTPS → mcp-proxy exits with error: "Error: MCP server URL must use HTTPS. Insecure HTTP connections are not allowed."
- **EXC-001d**: MCP server URL is unreachable → mcp-proxy exits with error: "Error: Failed to connect to MCP server at {url}: {error details}"
- **EXC-001e**: Port 3000 is already in use → mcp-proxy tries ports 3001-3010, if all fail exits with error: "Error: Cannot start OAuth callback server. Ports 3000-3010 are all in use."
- **EXC-001f**: User closes browser without completing OAuth → mcp-proxy waits 5 minutes then exits with error: "Error: OAuth flow timed out. User did not complete authentication."
- **EXC-001g**: OAuth token exchange fails → mcp-proxy exits with error: "Error: Failed to exchange authorization code for tokens: {error details}"
- **EXC-001h**: Cannot create ~/.cache/mcp-proxy/ directory → mcp-proxy exits with error: "Error: Cannot create token cache directory at ~/.cache/mcp-proxy/: {error details}"
- **EXC-001i**: Cannot write token file → mcp-proxy exits with error: "Error: Cannot save tokens to ~/.cache/mcp-proxy/{url_base64}.json: {error details}"

**Cross-scenario notes:** None (first-time use is independent)

### SC-002: Subsequent use with valid token
**Actor:** Developer

**Preconditions:**
- Token file exists at ~/.cache/mcp-proxy/{url_base64}.json
- access_token is not expired (current time < expiration_time in token file)
- Token file is valid JSON with required fields

**Flow:**
1. Developer runs: `mcp-proxy -u https://mcp.example.com`
2. mcp-proxy validates client_id and client_secret are not empty
3. mcp-proxy exposes stdio interface locally
4. mcp-proxy reads token file ~/.cache/mcp-proxy/{url_base64}.json
5. mcp-proxy parses JSON and validates required fields exist
6. mcp-proxy checks expiration_time, confirms access_token is still valid
7. mcp-proxy connects to remote MCP server via streamable HTTP with access_token in Authorization header
8. MCP server accepts the token and responds successfully
9. mcp-proxy forwards response to local client via stdio

**Postconditions:**
- No browser interaction occurred
- Token file unchanged
- Request succeeded immediately without authentication flow

**Exceptions:**
- **EXC-002a**: Token file is corrupted/invalid JSON → mcp-proxy treats as missing token, proceeds with SC-001 flow
- **EXC-002b**: Token file missing required fields → mcp-proxy treats as missing token, proceeds with SC-001 flow
- **EXC-002c**: MCP server rejects token (revoked server-side) → mcp-proxy receives 401, treats as expired token, proceeds with SC-003 flow (attempt refresh)
- **EXC-002d**: Network error connecting to MCP server → mcp-proxy exits with error: "Error: Failed to connect to MCP server: {error details}"

**Cross-scenario notes:** If token is rejected (EXC-002c), transitions to SC-003 (token refresh)

---

### SC-003: Token expired, refresh available
**Actor:** Developer

**Preconditions:**
- Token file exists at ~/.cache/mcp-proxy/{url_base64}.json
- access_token is expired (current time >= expiration_time)
- refresh_token exists in token file and is valid

**Flow:**
1. Developer runs: `mcp-proxy -u https://mcp.example.com`
2. mcp-proxy validates client_id and client_secret are not empty
3. mcp-proxy exposes stdio interface locally
4. mcp-proxy reads token file ~/.cache/mcp-proxy/{url_base64}.json
5. mcp-proxy checks expiration_time, detects access_token is expired
6. mcp-proxy uses refresh_token to request new access_token from OAuth2.1 token endpoint
7. OAuth2.1 server validates refresh_token and returns new access_token (and possibly new refresh_token)
8. mcp-proxy updates token file with new access_token, updated expiration_time, and new refresh_token (if provided)
9. mcp-proxy connects to remote MCP server via streamable HTTP with new access_token
10. MCP server responds successfully
11. mcp-proxy forwards response to local client via stdio

**Postconditions:**
- No browser interaction occurred
- Token file updated with fresh access_token and expiration_time
- refresh_token updated if server provided a new one
- Request succeeded after automatic refresh

**Exceptions:**
- **EXC-003a**: refresh_token is rejected by OAuth server (expired/revoked) → mcp-proxy proceeds with SC-004 flow (full re-authentication)
- **EXC-003b**: Network error during refresh request → mcp-proxy exits with error: "Error: Failed to refresh token: {error details}"
- **EXC-003c**: OAuth server is down/unreachable → mcp-proxy exits with error: "Error: Cannot reach OAuth server for token refresh: {error details}"
- **EXC-003d**: Cannot write updated token file → mcp-proxy exits with error: "Error: Token refresh succeeded but cannot save updated tokens: {error details}"

**Cross-scenario notes:** If refresh fails (EXC-003a), transitions to SC-004 (full re-authentication)

---

### SC-004: Token expired, no valid refresh
**Actor:** Developer

**Preconditions:**
- Token file exists at ~/.cache/mcp-proxy/{url_base64}.json
- access_token is expired (current time >= expiration_time)
- refresh_token is missing, expired, or invalid

**Flow:**
1. Developer runs: `mcp-proxy -u https://mcp.example.com`
2. mcp-proxy validates client_id and client_secret are not empty
3. mcp-proxy exposes stdio interface locally
4. mcp-proxy reads token file ~/.cache/mcp-proxy/{url_base64}.json
5. mcp-proxy checks expiration_time, detects access_token is expired
6. mcp-proxy attempts to use refresh_token to get new access_token
7. OAuth2.1 server rejects refresh_token (invalid/expired/revoked)
8. mcp-proxy discovers OAuth2.1 endpoints via .well-known/oauth-authorization-server
9. mcp-proxy starts temporary HTTP server on localhost:3000 for OAuth callback
10. mcp-proxy generates PKCE code_verifier and code_challenge
11. mcp-proxy opens system browser to OAuth2.1 authorization URL with callback http://localhost:3000/oauth2callback
12. User completes OAuth flow in browser (login, consent)
13. Browser redirects to localhost:3000/oauth2callback with authorization code
14. mcp-proxy exchanges authorization code for new access_token and refresh_token using PKCE code_verifier
15. mcp-proxy updates token file with new tokens and expiration_time
16. mcp-proxy stops temporary HTTP server
17. mcp-proxy connects to remote MCP server via streamable HTTP with new access_token
18. MCP server responds successfully
19. mcp-proxy forwards response to local client via stdio

**Postconditions:**
- Browser interaction occurred (user re-authenticated)
- Token file updated with fresh tokens
- Request succeeded after re-authentication

**Exceptions:**
- Same exceptions as SC-001 (EXC-001e through EXC-001i) apply during re-authentication

**Cross-scenario notes:** This scenario is triggered from SC-003 when refresh fails

## 6. Functional Requirements

### FR-001: CLI Argument Parsing
**Description:** Parse and validate command-line arguments for mcp-proxy configuration.

**Inputs:**
- `--url` or `-u`: MCP server URL (required)
- `--client-id` or `-i`: OAuth2.1 client ID (optional, default: env:GOOGLE_CLIENT_ID)
- `--client-secret` or `-s`: OAuth2.1 client secret (optional, default: env:GOOGLE_CLIENT_SECRET)

**Outputs:**
- Validated configuration object with resolved client_id, client_secret, and MCP server URL

**Business Rules:**
- URL flag is mandatory; exit with usage help if missing
- If client_id starts with "env:", read value from environment variable (e.g., "env:GOOGLE_CLIENT_ID" reads $GOOGLE_CLIENT_ID)
- If client_secret starts with "env:", read value from environment variable
- Default client_id is "env:GOOGLE_CLIENT_ID"
- Default client_secret is "env:GOOGLE_CLIENT_SECRET"
- If resolved client_id is empty, exit with clear error message
- If resolved client_secret is empty, exit with clear error message
- URL must start with "https://"; reject "http://" URLs with clear error message
- URL must be a valid URL format; reject malformed URLs with clear error message

**Priority:** Must-have

---

### FR-002: OAuth2.1 Discovery and Authentication
**Description:** Discover OAuth2.1 endpoints via .well-known mechanism and execute PKCE-enhanced authorization code flow.

**Inputs:**
- MCP server URL
- client_id and client_secret
- OAuth2.1 authorization response (authorization code from callback)

**Outputs:**
- access_token, refresh_token, expires_in from OAuth2.1 token endpoint

**Business Rules:**
- Fetch .well-known/oauth-authorization-server from MCP server URL
- Parse authorization_endpoint and token_endpoint from discovery document
- Generate cryptographically secure PKCE code_verifier (43-128 characters, base64url-encoded)
- Compute code_challenge as SHA256(code_verifier), base64url-encoded
- Start temporary HTTP server on localhost:3000 (or next available port 3001-3010)
- Construct authorization URL with: client_id, redirect_uri (http://localhost:{port}/oauth2callback), response_type=code, code_challenge, code_challenge_method=S256, scope (if required by server)
- Open system default browser to authorization URL
- Wait for callback with authorization code (timeout: 5 minutes)
- Exchange authorization code for tokens using: client_id, client_secret, code, redirect_uri, code_verifier, grant_type=authorization_code
- Stop temporary HTTP server after receiving callback
- Validate token response contains access_token and expires_in
- refresh_token is optional but should be stored if provided

**Priority:** Must-have

---

### FR-003: Token File Management
**Description:** Persist OAuth2.1 tokens to local filesystem with secure permissions and always read from disk (no in-memory caching).

**Inputs:**
- MCP server URL (for filename generation)
- access_token, refresh_token, expires_in from OAuth2.1 server

**Outputs:**
- Token file at ~/.cache/mcp-proxy/{url_base64}.json

**Business Rules:**
- Create ~/.cache/mcp-proxy/ directory if it doesn't exist (permissions: 0700)
- Filename is base64url-encoded MCP server URL with .json extension
- Token file format (JSON):
  ```json
  {
    "access_token": "...",
    "refresh_token": "...",
    "expiration_time": "2026-04-15T22:45:00Z"
  }
  ```
- expiration_time is computed as: current_time + expires_in (ISO 8601 UTC format)
- Token file permissions must be 0600 (read/write for owner only)
- Always read token file from disk on each mcp-proxy invocation (no in-memory caching across invocations)
- If token file read fails, treat as missing token and proceed with full OAuth flow
- If token file is invalid JSON, treat as missing token and proceed with full OAuth flow
- If token file write fails, exit with clear error message (do not proceed without persisting tokens)

**Priority:** Must-have

---

### FR-004: Token Expiration and Refresh
**Description:** Check token expiration before use and automatically refresh expired tokens using refresh_token.

**Inputs:**
- Token file contents (access_token, refresh_token, expiration_time)
- Current system time

**Outputs:**
- Valid access_token (either existing or refreshed)
- Updated token file (if refresh occurred)

**Business Rules:**
- Compare current system time with expiration_time from token file
- If current_time < expiration_time, use existing access_token
- If current_time >= expiration_time, attempt token refresh:
  - Use refresh_token to request new access_token from OAuth2.1 token endpoint
  - Request parameters: client_id, client_secret, refresh_token, grant_type=refresh_token
  - If refresh succeeds: update token file with new access_token, new expiration_time, and new refresh_token (if provided)
  - If refresh fails (401/400 response): proceed with full OAuth flow (SC-004)
- If refresh_token is missing from token file, proceed with full OAuth flow
- Token refresh should complete within 2 seconds (network timeout)

**Priority:** Must-have

---

### FR-005: MCP Server Proxy
**Description:** Expose stdio interface locally and proxy requests/responses to remote MCP server via streamable HTTP with OAuth2.1 authentication.

**Inputs:**
- stdin from local MCP client
- MCP server URL
- Valid access_token

**Outputs:**
- stdout to local MCP client (proxied responses from MCP server)

**Business Rules:**
- Read JSON-RPC messages from stdin (newline-delimited)
- Forward each message to MCP server via streamable HTTP POST request
- Include Authorization header: "Bearer {access_token}"
- Include Content-Type header: "application/json"
- Stream response from MCP server back to stdout
- Preserve message ordering (FIFO)
- If MCP server returns 401 Unauthorized, treat as token expired and attempt refresh (FR-004)
- If refresh succeeds, retry the failed request automatically
- If refresh fails, exit with error message
- Handle connection errors gracefully with clear error messages
- Request timeout: 30 seconds per request

**Priority:** Must-have

---

### FR-006: Error Handling and User Feedback
**Description:** Provide clear, actionable error messages for all failure scenarios.

**Inputs:**
- Error conditions from any component (CLI parsing, OAuth flow, token management, MCP proxy)

**Outputs:**
- Human-readable error messages to stderr
- Non-zero exit codes

**Business Rules:**
- All error messages must start with "Error: "
- Error messages must be specific and actionable (tell user what went wrong and how to fix it)
- Never include sensitive data in error messages (tokens, secrets, authorization codes)
- Exit codes:
  - 0: Success
  - 1: Configuration error (missing/invalid arguments)
  - 2: Authentication error (OAuth flow failed)
  - 3: Network error (cannot reach MCP server or OAuth server)
  - 4: File system error (cannot read/write token file)
  - 5: Token error (token expired and refresh failed)
- Log errors to OpenTelemetry traces with ERROR level
- For network errors, include timeout information if applicable
- For OAuth errors, include error code and description from OAuth server (but not sensitive details)

**Priority:** Must-have

## 7. Non-Functional Requirements

### 7.1 Performance
- **Startup time:** < 1 second from command invocation to stdio interface ready
- **Token refresh:** < 2 seconds to complete refresh_token exchange
- **OAuth flow:** No hard limit (depends on user interaction), but timeout after 5 minutes of inactivity
- **MCP request forwarding:** < 100ms latency overhead (excluding network time to MCP server)
- **Token file read:** < 10ms
- **Token file write:** < 50ms

### 7.2 Security
- **HTTPS enforcement:** Reject any MCP server URL that uses http:// (only https:// allowed)
- **PKCE:** Use PKCE (Proof Key for Code Exchange) for all OAuth2.1 authorization flows
- **Token storage:** Store tokens in ~/.cache/mcp-proxy/ with file permissions 0600 (owner read/write only)
- **Directory permissions:** ~/.cache/mcp-proxy/ directory must have permissions 0700 (owner read/write/execute only)
- **Sensitive data in logs:** Never log tokens, client secrets, authorization codes, or PKCE code_verifier
- **Sensitive data in errors:** Never include tokens or secrets in error messages
- **OAuth callback server:** Temporary HTTP server only listens on localhost (127.0.0.1), not on all interfaces
- **OAuth callback server lifecycle:** Start only when needed, stop immediately after receiving callback
- **Token transmission:** Always use HTTPS for communication with OAuth server and MCP server
- **Code_verifier generation:** Use cryptographically secure random number generator

### 7.3 Usability
- **Clear error messages:** All errors must be actionable and user-friendly
- **Minimal configuration:** Only 3 CLI flags, with sensible defaults for Google OAuth
- **Automatic token management:** No manual token refresh or re-authentication required
- **Browser integration:** Automatically open system default browser for OAuth flow
- **Cross-platform:** Support Linux, macOS, and Windows
- **No external dependencies:** Single binary with no runtime dependencies

### 7.4 Reliability
- **Token persistence:** Tokens must survive mcp-proxy restarts
- **Graceful degradation:** If token refresh fails, fall back to full OAuth flow
- **Idempotent operations:** Token file writes must be atomic (write to temp file, then rename)
- **Error recovery:** Clear error messages guide user to resolution
- **Network resilience:** Retry logic for transient network errors (up to 3 retries with exponential backoff)

### 7.5 Observability
OpenTelemetry tracing is mandatory for all operations.

- **Collector:** JSONL file output (default, no external dependencies)
- **Trace file path:** ~/.cache/mcp-proxy/traces.jsonl
- **Trace file permissions:** 0600 (owner read/write only)
- **Trace file rotation:** Append-only, no automatic rotation (user can manually truncate/delete)
- **What to trace:**
  - OAuth flow steps (discovery, authorization, token exchange, refresh) at INFO level
  - MCP server HTTP requests/responses (method, URL, status code, duration) at INFO level
  - Token file operations (read, write, validation) at DEBUG level
  - Errors and warnings at ERROR/WARNING level
  - PKCE generation and validation at DEBUG level
  - Temporary HTTP server lifecycle (start, stop, callback received) at INFO level
- **Sensitive data exclusion:** NEVER trace:
  - access_token, refresh_token, authorization codes
  - client_secret
  - PKCE code_verifier
  - Full request/response bodies (only metadata like status codes, content-length)
- **Trace attributes to include:**
  - mcp.server.url (MCP server URL, sanitized)
  - oauth.flow.step (discovery, authorization, token_exchange, refresh)
  - http.method, http.status_code, http.duration_ms
  - error.type, error.message (sanitized, no sensitive data)
  - token.expiration_time (for debugging expiration logic)
  - token.refresh_attempted (boolean)

### 7.6 Deployment
- **Platform:** CLI tool, runs locally on developer's machine
- **Distribution:** GitHub Releases with pre-built binaries
  - Linux: amd64, arm64
  - macOS: amd64 (Intel), arm64 (Apple Silicon)
  - Windows: amd64
- **Binary naming:** mcp-proxy (Linux/macOS), mcp-proxy.exe (Windows)
- **Installation:** Manual download from GitHub Releases, place in PATH
- **No installation script:** Users manually download and place binary in PATH
- **Versioning:** Semantic versioning (MAJOR.MINOR.PATCH)
- **Release artifacts:** Compressed archives (.tar.gz for Linux/macOS, .zip for Windows)
- **Checksums:** SHA256 checksums for all release artifacts

### 7.7 Scalability
- **Single-user:** Designed for single developer use (no multi-user support)
- **Single MCP server:** One mcp-proxy instance per MCP server URL
- **Concurrent instances:** Multiple mcp-proxy instances can run simultaneously for different MCP servers
- **Token file isolation:** Each MCP server URL has its own token file (no conflicts)
- **Resource usage:** Minimal memory footprint (< 50MB), minimal CPU usage when idle

## 8. Data Model

### Token File Structure

**Location:** `~/.cache/mcp-proxy/{url_base64}.json`

**Format:** JSON

**Schema:**
```json
{
  "access_token": "string (OAuth2.1 access token)",
  "refresh_token": "string (OAuth2.1 refresh token, optional)",
  "expiration_time": "string (ISO 8601 UTC timestamp, e.g., 2026-04-15T22:45:00Z)"
}
```

**Field Descriptions:**
- `access_token`: Bearer token for authenticating with MCP server. Required.
- `refresh_token`: Token for refreshing expired access_token. Optional (some OAuth servers don't provide it).
- `expiration_time`: Absolute timestamp when access_token expires. Computed as current_time + expires_in from OAuth response. Required.

**File Permissions:** 0600 (owner read/write only)

**Filename Generation:**
- Take MCP server URL (e.g., "https://mcp.example.com")
- Base64url-encode the URL (RFC 4648, URL-safe alphabet, no padding)
- Append ".json" extension
- Example: "https://mcp.example.com" → "aHR0cHM6Ly9tY3AuZXhhbXBsZS5jb20.json"

**Relationships:**
- One token file per MCP server URL
- Token files are independent (no shared state)

## 12. End-to-End Test Suite

> **This is the most important section of the specification.** E2E tests are the primary contract for agent-based implementation. An agent will use these tests as its definition of "done": the implementation is correct when all E2E tests pass. Each test must be precise enough that an agent can write both the test code and the production code from it alone.

All tests MUST be implemented in the `tests/` directory using Go's testing framework with table-driven tests. Each feature MUST have tests covering happy paths, failure paths, side effects, edge cases, and error recovery. **Failure tests must outnumber happy-path tests.**

## 16. Glossary

- **MCP (Model Context Protocol)**: A protocol for communication between AI assistants and external tools/services
- **OAuth2.1**: The latest version of the OAuth 2.0 authorization framework
- **PKCE (Proof Key for Code Exchange)**: An OAuth extension that prevents authorization code interception attacks
- **Streamable HTTP**: HTTP-based transport that supports streaming responses
- **stdio**: Standard input/output, a communication method using stdin and stdout
- **access_token**: A credential used to access protected resources, typically short-lived
- **refresh_token**: A credential used to obtain new access tokens without user interaction, typically long-lived
- **code_verifier**: A cryptographically random string used in PKCE flow
- **code_challenge**: SHA256 hash of code_verifier, sent in authorization request
- **.well-known**: A standard path for service discovery documents (RFC 8414)
- **base64url**: URL-safe base64 encoding (RFC 4648) without padding
- **ISO 8601**: International standard for date and time representation
- **OpenTelemetry**: An observability framework for generating, collecting, and exporting telemetry data
- **JSONL**: JSON Lines format, one JSON object per line
